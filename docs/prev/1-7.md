# Báo Cáo Phiên Làm Việc Ngày 01/07/2026 — Hoàn Thiện Cloud File Storage

Tài liệu này tổng hợp toàn bộ các quyết định kiến trúc, thay đổi mã nguồn và hướng dẫn dành cho các agent sau này, tập trung vào việc chuyển đổi hoàn toàn cơ chế lưu trữ tài liệu từ Local VPS sang Cloudinary (từ Phase 1 đến Phase 5).

## 1. Các Quyết Định Kiến Trúc Trọng Tâm

### 1.1. Phân Tách Nhiệm Vụ Giữa `storageKey` và `publicUrl`

- **`storageKey`**: Là định danh nội bộ (private/internal) dùng riêng cho Backend, Worker và các script dọn dẹp để định vị tệp.
  - **KHÔNG BAO GIỜ** được trả về cho Frontend (bị loại bỏ khỏi mọi API response: upload, status, detail, download).
  - Với Cloudinary, định dạng mới được sử dụng là `resourceType:publicId` (vd: `raw:documents/abc.pdf`). Tuy nhiên, adapter vẫn hỗ trợ định dạng legacy secure URL để đảm bảo tương thích ngược (backward compatible) với các tệp cũ.
- **`publicUrl`**: Là URL chia sẻ công khai của hệ thống, chỉ có giá trị khi tài liệu được gán cờ `isPublic = true`.
  - **KHÔNG BAO GIỜ** sử dụng Cloudinary secure URL để gán cho `publicUrl`. Điều này nhằm ngăn chặn lỗi bảo mật bypass phân quyền.
  - Đối với tệp được chia sẻ, `publicUrl` sẽ trỏ đến URL của ứng dụng theo định dạng `/shared/:token`.

### 1.2. Download Qua App Proxy (Không Redirect)

Endpoint `GET /documents/:id/download` không còn redirect trực tiếp sang Cloudinary URL.
Luồng tải xuống chuẩn:

1. Backend kiểm tra quyền truy cập của người dùng.
2. Backend gọi storage adapter để lấy stream trực tiếp từ Cloudinary hoặc Local disk.
3. Backend pipe trực tiếp stream này về response cho Client.

## 2. Chi Tiết Triển Khai Theo Phase

### Phase 1: Storage Adapter & Cloudinary SDK

- **Mục tiêu**: Xây dựng layer trừu tượng cho hệ thống lưu trữ.
- **Thực hiện**:
  - `UploadResult` đã loại bỏ trường `publicUrl`.
  - `CloudinaryStorage` tự động sinh và xử lý `storageKey` theo chuẩn mới (`resourceType:publicId`), đồng thời tự build internal delivery URL qua hàm `getDownloadUrl()`. Hàm `delete()` hỗ trợ cả format mới và legacy.
  - Tích hợp Factory Pattern để tự động chọn `CloudinaryStorage` hoặc `LocalStorage` dựa trên cấu hình môi trường (`STORAGE_PROVIDER`).

### Phase 2: Flow Quản Lý Document & Sharing (Phase 2 & 2b)

- **Upload/Update**:
  - Tệp tải lên thông qua adapter, lưu lại `storageProvider`, `storageBucket`, và `storageKey`. `publicUrl` mặc định bằng rỗng (`''`).
  - Xóa tệp tạm (temp local file) ngay khi upload cloud thành công. Tự động xóa file trên Cloudinary (rollback) nếu quá trình lưu DB thất bại.
  - Cờ `isPublic` khi cập nhật sẽ kích hoạt event cập nhật URL tương ứng.
- **Download/Status**:
  - Proxy stream hoàn toàn qua Backend.
  - API `getUploadStatus()` và `getDocumentDetail()` được thiết kế để ẩn `storageKey`.
- **Share Link**:
  - Tạo share link gán `publicUrl` = `/shared/:token`.
  - Bổ sung endpoint `GET /shared/:token/file` hỗ trợ kiểm tra tính hợp lệ của token, giới hạn số lượt tải, sau đó stream file trực tiếp.

### Phase 3: Migration & Dọn Dẹp (Hard-Delete)

- **Migration** (`scripts/migrate-local-to-cloud.ts`): Cập nhật Collection `solutions` và `document_extraction_jobs` thành storage metadata mới mà không làm thay đổi các giá trị `publicUrl` hiện tại của người dùng.
- **Cleanup** (`scripts/cleanup-deleted-documents.ts`): Script dọn dẹp các tệp bị xóa vĩnh viễn ưu tiên sử dụng `storageProvider`, có fallback sang legacy URL nếu cần.

### Phase 4 & 5: Worker Extraction và Avatar

- **Worker Extraction**: `DocumentExtractionJob` được bổ sung metadata lưu trữ (`storageProvider`, `storageBucket`). Worker không còn dùng `fs` mà sử dụng hàm `getFileStream()` từ storage adapter và chuyển đổi thành stream/Buffer giả lập Multer.File để trích xuất văn bản.
- **Avatar Upload**: Profile người dùng có thêm các trường lưu trữ nội bộ (ẩn khỏi API). Upload avatar sử dụng adapter (vào thư mục `avatars`, `resourceType = image`). URL avatar là đường dẫn thumbnail đã optimize.
- **Deploy**: Cấu hình `compose.cloud.yaml` mới (không mount volume uploads) và giữ lại volume trong `compose.yaml` gốc làm fallback.

## 5. Cẩm Nang Cho Các Phiên Agent Sau (Handoff Notes)

Để bảo đảm tính ổn định và bảo mật của toàn bộ hệ thống lưu trữ, các agent làm việc sau vui lòng tuân thủ tuyệt đối 5 quy tắc sau:

1. **KHÔNG BAO GIỜ gán `publicUrl = storageKey`**: Đây là lỗ hổng bảo mật rò rỉ URL file gốc. Mọi tài liệu cần xem/tải đều phải đi qua Proxy route của ứng dụng để kiểm tra phân quyền rồi mới stream dữ liệu.
2. **KHÔNG BAO GIỜ trả `storageKey` ra Frontend**: Ẩn triệt để thuộc tính này trong mọi endpoint response (detail, status, upload, download).
3. **Internal File Access**: Mọi Service và Worker nội bộ BẮT BUỘC đọc file thông qua tổ hợp `storageProvider + storageKey` từ Storage Adapter (KHÔNG dùng `publicUrl`).
4. **Quy tắc định dạng Cloudinary Key**: Hệ thống đã chuyển sang dùng cấu trúc `resourceType:publicId`. Adapter tự động hỗ trợ dịch ngược URL cũ, KHÔNG nên sửa đổi để tránh vỡ dữ liệu.
5. **Cơ chế Temporary Uploads**: Khởi tạo Multer luôn cần ghi tạm ra thư mục `uploads/` bên trong container. Do đó, KHÔNG xóa cấu hình tạo thư mục `uploads/` trong Dockerfile kể cả khi đã triển khai Cloud.
